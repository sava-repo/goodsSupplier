## ADDED Requirements

### Requirement: Создание поставщика
Система SHALL позволять создавать нового поставщика с обязательными полями: название, город. Остальные поля опциональны: описание, контактное лицо, телефон, email, сайт, источник информации, регион, адрес, минимальный объём заказа, диапазон цен, наличие сертификатов, детали сертификатов, условия доставки, заметки.

#### Scenario: Успешное создание поставщика с обязательными полями
- **WHEN** пользователь отправляет POST запрос на `/api/suppliers` с полями `name`="ООО Свежие Продукты" и `city`="Москва"
- **THEN** система создаёт поставщика и возвращает статус 201 с данными созданного поставщика, включая сгенерированный `id`

#### Scenario: Создание поставщика без обязательного поля
- **WHEN** пользователь отправляет POST запрос на `/api/suppliers` без поля `name`
- **THEN** система возвращает статус 400 с описанием ошибки валидации

#### Scenario: Создание поставщика с полной информацией
- **WHEN** пользователь отправляет POST запрос со всеми полями, включая категории
- **THEN** система создаёт поставщика со всеми полями и связями с категориями, возвращает статус 201

### Requirement: Получение списка поставщиков
Система SHALL возвращать список всех активных поставщиков с базовыми полями: id, название, город, категории, диапазон цен. Система MUST поддерживать пагинацию с параметрами `page` и `per_page` (по умолчанию 20 на страницу).

#### Scenario: Получение первой страницы поставщиков
- **WHEN** пользователь отправляет GET запрос на `/api/suppliers`
- **THEN** система возвращает список поставщиков (до 20 штук), общее количество и информацию о пагинации

#### Scenario: Получение конкретной страницы
- **WHEN** пользователь отправляет GET запрос на `/api/suppliers?page=2&per_page=10`
- **THEN** система возвращает вторую страницу с 10 поставщиками и корректную метаинформацию о пагинации

### Requirement: Получение детальной информации о поставщике
Система SHALL возвращать полную информацию о поставщике по его идентификатору, включая все поля и связанные категории.

#### Scenario: Поставщик существует
- **WHEN** пользователь отправляет GET запрос на `/api/suppliers/1`
- **THEN** система возвращает все данные поставщика с id=1, включая список категорий

#### Scenario: Поставщик не найден
- **WHEN** пользователь отправляет GET запрос на `/api/suppliers/999`
- **THEN** система возвращает статус 404 с сообщением об ошибке

### Requirement: Обновление поставщика
Система SHALL позволять обновлять любые поля поставщика по его идентификатору.

#### Scenario: Успешное обновление полей
- **WHEN** пользователь отправляет PUT запрос на `/api/suppliers/1` с обновлённым полем `phone`
- **THEN** система обновляет поле и возвращает полные обновлённые данные поставщика

#### Scenario: Обновление несуществующего поставщика
- **WHEN** пользователь отправляет PUT запрос на `/api/suppliers/999`
- **THEN** система возвращает статус 404

### Requirement: Удаление поставщика
Система SHALL позволять удалять поставщика по его идентификатору. Удаление SHALL быть мягким (установка `is_active` = false).

#### Scenario: Успешное удаление
- **WHEN** пользователь отправляет DELETE запрос на `/api/suppliers/1`
- **THEN** система устанавливает `is_active` = false для поставщика и возвращает статус 200

#### Scenario: Удаление несуществующего поставщика
- **WHEN** пользователь отправляет DELETE запрос на `/api/suppliers/999`
- **THEN** система возвращает статус 404

### Requirement: Связь поставщика с категориями
Система SHALL позволять связывать поставщика с одной или несколькими категориями товаров. Один поставщик MAY относиться к нескольким категориям.

#### Scenario: Создание поставщика с категориями
- **WHEN** пользователь создаёт поставщика с `category_ids` = [1, 3]
- **THEN** система создаёт поставщика и устанавливает связь с категориями id=1 и id=3

#### Scenario: Обновление категорий поставщика
- **WHEN** пользователь обновляет поставщика с новым набором `category_ids` = [2, 4]
- **THEN** система заменяет старые связи на новые (категории 1 и 3 удаляются, добавляются 2 и 4)
